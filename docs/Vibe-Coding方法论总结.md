# Vibe Coding 方法论总结

> 基于「后真相时代」信息甄别平台开发全程的实践提炼

---

## 一、Prompt 工程经验

### 1.1 结构化需求描述法

**核心原则：将模糊需求转化为可执行的技术规格**

在本项目开发过程中，最有效的 Prompt 模式是"**上下文 + 目标 + 约束 + 验收标准**"四段式结构：

```
[上下文] 我们正在开发一个新闻核实平台，后端使用 FastAPI + SQLAlchemy async
[目标]   实现 RSS 数据源的自动采集，并通过 NLP 将文章归类到对应的舆论案例
[约束]   需要支持中文，已安装 jieba 和 sentence-transformers
[验收]   采集后 last_collected_at 字段更新，同一 URL 不重复插入
```

**实际效果对比**：

| Prompt 类型 | 示例 | AI响应质量 | 需要补充次数 |
|------------|------|-----------|------------|
| 模糊型 | "帮我做一个采集功能" | 需大量补充 | 3-5 次 |
| 结构化型 | 四段式描述 | 直接可用 | 0-1 次 |

---

### 1.2 分步拆解法（Incremental Decomposition）

**适用场景**：功能复杂、涉及多文件修改时

本项目 Feature 004（智能事件归类）涉及后端 NLP 管线、前端颜色修复、RSS 源替换等 15 个子任务，直接提出"实现智能归类"会导致 AI 产出不完整代码。正确做法是逐层拆解：

```
第一层：功能拆解
"智能事件归类" → 实体提取模块 + 语义匹配模块 + 融合分类逻辑 + 缓存机制

第二层：单任务实现
"实现 entity_extractor.py：
  - 使用 jieba.posseg 提取人名/地名/机构名
  - 过滤停用词和单字词
  - 返回 set[str] 用于后续 Jaccard 计算"

第三层：集成验证
"为 entity_extractor.py 编写测试用例，覆盖
  空文本/纯停用词/含命名实体/重叠度计算 4 个场景"
```

**经验规律**：单个 Prompt 的最优粒度是"一个函数或一个组件，包含完整的输入输出定义"。过大会导致遗漏，过小则割裂上下文。

---

### 1.3 错误驱动迭代法（Error-Driven Iteration）

**核心技巧：将报错信息完整喂给 AI，而不是转述**

本项目多次遇到非直觉错误，例如：

- **SQLite 时区丢失问题**：`datetime.now(timezone.utc)` 存入 SQLite 后 `+00:00` 丢失，前端显示差 8 小时
- **字段名映射错误**：后端返回 `source_name` 但前端类型定义仍为 `name`

处理方式：不描述"时间显示不对"，而是直接提供：
1. 数据库实际存储值 `2026-03-11 10:30:04.498510`（无时区）
2. API 返回 JSON `"last_collected_at": "2026-03-11T10:30:04"`（无 Z）
3. 浏览器 `new Date("2026-03-11T10:30:04")` 解析结果（当本地时间）

完整的证据链使 AI 能准确定位根因（Pydantic 序列化时无时区信息），给出精准的 `model_validator` 修复方案，而非猜测性方案。

---

### 1.4 上下文锚定法（Context Anchoring）

**问题**：长对话后 AI 容易遗忘早期设计决策，导致新代码与已有代码风格/接口不一致。

**解决方案**：在关键节点建立"锚点 Prompt"，明确声明已有约定：

```
"注意以下已有约定（不要改动）：
  - 事件可信度等级用 CredibilityLevel 枚举，共 4 值
  - 所有时区统一使用 UTC，前端转换为本地时间
  - Vue 组件使用 <script setup> + Composition API
  - API 路径前缀为 /api/v1/"
```

在每次涉及跨文件修改的 Prompt 开头加入此类锚点，可将接口不一致错误降低约 70%。

---

### 1.5 反事实验证法（Counterfactual Verification）

**适用场景**：对 AI 给出的算法或架构决策存疑时

在设计双层 NLP 分类时，通过向 AI 提问反向问题来验证方案合理性：

```
"为什么不直接用纯语义相似度？实体重叠的价值在哪里？"
"如果语义模型加载失败，系统会怎样？当前设计有降级路径吗？"
"Jaccard 系数在实体集合很小时（只有1-2个词）会不会过于严格？"
```

通过追问"如果不这样会怎样"，迫使 AI 显式论证设计决策，暴露潜在缺陷，同时也加深了对方案的理解。

## 二、AI 辅助编程效率分析

### 2.1 开发效率量化对比

本项目总代码量（不含测试）约 **4,200 行**，功能覆盖全栈 8 模块。传统开发 vs. Vibe Coding 估算对比：

| 模块 | 传统开发估时 | Vibe Coding 实际耗时 | 提效倍数 |
|------|------------|---------------------|---------|
| 数据库模型 + 迁移（8 表） | 3 天 | 2 小时 | ~12× |
| REST API 全部端点（11 个） | 5 天 | 4 小时 | ~10× |
| NLP 双层分类引擎 | 2 周 | 6 小时 | ~14× |
| 三维可信度评估模型 | 1 周 | 3 小时 | ~14× |
| Vue 3 前端全部页面 | 2 周 | 8 小时 | ~12× |
| 测试套件（117 个用例） | 1 周 | 3 小时 | ~14× |
| **合计** | **~6 周** | **~26 小时** | **~13×** |

> 注：传统开发估时基于有经验的全栈工程师独立完成，Vibe Coding 耗时含阅读/验证/调试。

---

### 2.2 AI 工具的核心优势

**① 跨领域知识即时调用**

本项目横跨 NLP（jieba/sentence-transformers）、异步 Python（FastAPI/SQLAlchemy async）、Vue 3 Composition API 三个专业领域。传统开发需要分别查阅文档、学习 API；AI 能直接输出符合最佳实践的代码片段，将学习成本压缩至近乎为零。

典型案例：Pydantic v2 的 `model_validator(mode="after")` 用法、APScheduler `misfire_grace_time` 配置、Vue 3 `defineEmits` 类型安全写法——这些都是文档散落在各处的"隐性知识"，AI 直接给出了正确用法。

**② 样板代码消除**

CRUD 端点、测试 Fixture、TypeScript 类型定义这类高度重复的样板代码，AI 能在 30 秒内生成，且质量稳定。测试套件（117 个用例）几乎完全由 AI 生成，人工只需验证覆盖场景是否合理。

**③ 错误诊断速度**

复杂 bug（如 SQLite 时区丢失、Vue 组件事件名不匹配、NLP 模型 O(n²) 性能问题）传统排查需要 1-4 小时。将完整的日志 + 堆栈 + 相关代码提供给 AI，通常 5 分钟内能定位根因并给出修复方案。

---

### 2.3 AI 工具的局限性

**① 上下文窗口瓶颈**

当项目代码量超过一定规模，AI 无法同时"看到"所有相关文件。本项目中曾出现 AI 修改了 `CredibilityPanel.vue` 的列 key，却不知道 `types/index.ts` 中的 `SourceWithCredibility` 接口也需要同步修改，导致运行时字段名不一致。

解决策略：在 Prompt 中主动列出所有相关文件路径，让 AI 先读取后再修改。

**② 无法验证业务正确性**

AI 能保证代码语法正确、逻辑自洽，但无法验证业务规则是否符合真实需求。例如可信度评估中"时效性每天衰减 5 分"是否合理、"交叉验证满分阈值 ≥5 个来源"是否过高——这些需要领域知识和用户验证，AI 只是实现了指令，不会主动质疑业务合理性。

**③ 新兴技术滞后**

AI 对高版本库（如本项目的 FastAPI 0.135、Pydantic v2 `model_validator`）有时会给出旧版 API 写法（如 Pydantic v1 的 `@validator`），需要手动纠正。经验：使用新库时应在 Prompt 中明确声明版本，如"使用 Pydantic v2，不要用 @validator，使用 @field_validator"。

**④ 缺乏主动架构建议**

AI 倾向于"被动执行"：你让它实现什么就实现什么，很少主动指出"这个设计可能有扩展性问题"。例如当前的 `_embedding_cache` 是进程内内存缓存，多进程部署时会失效——AI 实现了缓存，但没有主动提示这个限制。

## 三、改进空间与反思

### 3.1 当前项目的不足

**① 认证系统缺失**

当前所有 API 端点均无鉴权保护（数据源管理、事件创建等写操作均可匿名调用）。初期设计了 JWT 鉴权框架（`python-jose`、`passlib` 已在依赖中），但在简化导航时连同登录页一并移除，形成安全退步。生产环境不可在此状态下部署。

**② 评分阈值缺乏数据校验**

双层分类阈值（实体重叠 0.3、语义相似 0.45）和可信度评分权重（权威 40%、时效 20%、交叉 40%）均为经验设定，未经过真实数据集的系统性验证。可能存在对特定新闻类型（如技术/科学新闻，实体词汇稀少）分类准确率低的问题。

**③ 单体架构扩展性**

采集、NLP 推理、数据库写入均在同一 uvicorn 进程中同步执行。当采集量增大时（如扩展到 50+ 个新闻源），NLP 推理将阻塞其他请求，需要解耦为消息队列（Celery/RQ）+ Worker 模式。

**④ 前端无加载状态保护**

部分页面切换时存在短暂的空内容闪烁（数据未加载完成时无 skeleton 占位），Credibility 抽屉面板在数据加载期间无 loading 指示，影响体验连续性。

---

### 3.2 如有更多时间的优化方向

**短期（1 周内可完成）**

| 优化项 | 预期收益 |
|--------|---------|
| 恢复基于 API Key 的简单鉴权 | 保护管理接口的写操作安全 |
| 增加前端 Skeleton 加载占位 | 消除页面闪烁，提升体验连续性 |
| 将 NLP 阈值提取为可配置项并加 UI 面板 | 管理员可动态调整归类灵敏度 |
| 为 8 个 RSS 源添加连通性测试功能 | 快速定位失效源 |

**中期（1 个月内）**

| 优化项 | 预期收益 |
|--------|---------|
| 引入 Celery + Redis 解耦采集任务 | 支持高并发采集，不阻塞 API 响应 |
| 添加用户行为分析（阅读/收藏/举报） | 数据积累为个性化推荐奠基 |
| 基于用户举报的可信度动态修正 | 引入群体智慧，提升评分准确性 |
| 在 CI/CD 中加入 vue-tsc + ruff 检查 | 自动化代码质量门禁 |

**长期（2-3 个月）**

| 优化项 | 预期收益 |
|--------|---------|
| 用标注数据集评估并微调分类阈值 | 将分类准确率从经验估算提升到数据驱动 |
| 接入多模态分析（图片/视频存档） | 扩展核实能力至非文本信息 |
| 基于用户画像的个性化案例推荐 | 提升平台用户粘性 |
| 将可信度模型开放为独立 API | 扩展项目功能边界，支持第三方集成 |

---

### 3.3 方法论反思

**最有价值的实践**：维护一个共享的"约定清单"（数据库表名、API 路径规范、类型命名规则）。每次启动新对话时将其作为系统上下文注入，是解决 AI 上下文遗忘问题最有效的手段。

**最大的教训**：不要在同一个 Prompt 中同时要求"实现功能"和"修复另一个 Bug"。这会导致 AI 在两者之间权衡，产出代码往往两者都不彻底。严格的"一个 Prompt 一个任务"原则，是保证代码质量的基础。

**对 Vibe Coding 本质的理解**：AI 不是替代开发者思考，而是将开发者的认知带宽从"怎么写代码"解放到"写什么、为什么这样写"。开发者的核心工作从编码转向了系统设计、需求拆解、和验收判断——这恰好是最需要人类判断力的环节。
